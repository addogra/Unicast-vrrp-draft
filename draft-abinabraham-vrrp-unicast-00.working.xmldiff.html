67    <abstract>
68      <t>
69        The Virtual Router Redundancy Protocol (VRRP) is specified for
70        multicast operation on a shared LAN in RFC 9568. Some deployments,
71        including virtualized and cloud environments, require VRRP-like
72        first-hop redundancy but cannot use multicast delivery for VRRP
73        advertisements. This document updates RFC 9568 by defining an
74        optional unicast mode for VRRP.
75      </t>
76      <t>
77        In unicast mode, VRRP advertisements are sent to configured peer
78        addresses rather than to the VRRP multicast group. This document
79        preserves the VRRP packet format, protocol number, virtual IP
80        semantics, and host-facing forwarding behavior defined in RFC 9568,
81        while adding explicit peer configuration and receive-side source
82        validation for unicast operation.
83      </t>
84    </abstract>

88    <section anchor="intro">
89      <name>Introduction</name>
90      <t>
91        <xref target="RFC9568"/> specifies VRRP for IPv4 and IPv6 and assumes
92        multicast operation on a shared LAN. In a number of modern
93        deployments, redundant routers still need fast active/backup failover
94        for a virtual default gateway, but the environment does not provide
95        usable multicast support for VRRP advertisements.
96      </t>
97      <t>
98        The primary deployment driver is the continued need for the classic
99        VRRP function of protecting a virtual IPv4 or IPv6 first-hop gateway
100        in environments where multicast delivery is unavailable, undesirable,
101        or operationally constrained. This includes virtualized, cloud,
102        overlay, and other deployments in which Virtual Routers still provide
103        a common host-facing gateway service, but control traffic between the
104        Virtual Routers is exchanged through explicit peer connectivity
105        instead of a simple multicast-capable LAN.
106      </t>
107      <t>
108        The intended use case for this document is not a generic active/backup
109        role-election mechanism. Rather, it is a narrow extension of VRRP for
110        deployments that want to preserve the familiar VRRP state machine,
111        protocol number, virtual IP address semantics, Virtual Router MAC
112        behavior, and host-facing forwarding model while replacing only the
113        advertisement delivery method.
114      </t>

131    <section anchor="scope">
132      <name>Scope and Applicability</name>
133      <t>
134        This document updates <xref target="RFC9568"/> by defining an
135        optional unicast mode of operation for VRRP. The unicast mode is
136        intended for deployments that still want the classic VRRP model of a
137        Virtual Router protecting one or more virtual IPv4 or IPv6 addresses,
138        but that cannot rely on multicast delivery of advertisements.
139      </t>
140      <t>
141        The unicast mode defined here is limited to deployments in which the
142        participating VRRP Routers can exchange advertisements without
143        traversing a router that decrements the IPv4 TTL or IPv6 Hop Limit.
144        This preserves the receive-side validation model of
145        <xref target="RFC9568"/> and keeps the security and topology
146        assumptions close to those of the base protocol.
147      </t>
148      <t>
149        This document does not define multi-hop operation. If a deployment
150        requires routed multi-hop active/backup election or transport
151        encapsulation other than IP protocol 112, that deployment is outside
152        the scope of this specification.
153      </t>
154    </section>

156    <section anchor="usecases">
157      <name>Use Cases and Deployment Drivers</name>
158      <t>
159        The operational motivation for unicast VRRP is straightforward:
160        operators still need the classic VRRP function of protecting a virtual
161        IPv4 or IPv6 default gateway, but some modern deployments cannot rely
162        on multicast delivery for VRRP advertisements.
163      </t>
164      <t>
165        Typical use cases include:
166      </t>
167      <ol spacing="normal" type="1">
168        <li>
169          <t>
170            Cloud and virtualized environments in which multicast is not
171            available or is not exposed in a way that is operationally
172            equivalent to a simple shared LAN.
173          </t>
174        </li>
175        <li>
176          <t>
177            Overlay or virtualized edge deployments in which the protected
178            routers still present a common host-facing gateway service, but the
179            control traffic between the routers is exchanged using explicit
180            peer connectivity rather than a multicast-capable LAN.
181          </t>
182        </li>
183        <li>
184          <t>
185            Service-provider and data-center designs in which operators want to
186            preserve the familiar VRRP packet format, state machine, and
187            virtual IP semantics of <xref target="RFC9568"/> while replacing
188            only the advertisement delivery method.
189          </t>
190        </li>
191      </ol>
192      <t>
193        This document addresses those use cases by defining a unicast delivery
194        mode that stays close to the original VRRP model. It does not attempt
195        to standardize broader routed active/backup election mechanisms that
196        are no longer centered on protecting a virtual first-hop gateway.
197      </t>
198    </section>

229    <section anchor="overview">
230      <name>Unicast VRRP Overview</name>
231      <t>
232        A Virtual Router defined by this document operates in exactly one of
233        two modes:
234      </t>
235      <ul spacing="normal">
236        <li>multicast mode, as specified in <xref target="RFC9568"/>, or</li>
237        <li>unicast mode, as specified in this document.</li>
238      </ul>
239      <t>
240        A VRRP Router operating a given Virtual Router in unicast mode
241        <bcp14>MUST NOT</bcp14> send VRRP advertisements for that Virtual
242        Router to the VRRP multicast destination address. Instead, it
243        <bcp14>MUST</bcp14> send a copy of each advertisement to each address
244        in the configured Unicast Peer List.
245      </t>
246      <t>
247        Except as updated by this document, the VRRP packet format, VRRP
248        state machine, timer calculations, preemption behavior, Virtual Router
249        semantics, virtual IP address behavior, and host-facing forwarding
250        behavior remain as specified in <xref target="RFC9568"/>.
251      </t>
252      <t>
253        Unicast mode is configured per Virtual Router and per address family.
254        A VRRP Router <bcp14>MUST NOT</bcp14> mix multicast mode and unicast
255        mode for the same Virtual Router instance.
256      </t>
257    </section>

259    <section anchor="peer-config">
260      <name>Peer Configuration</name>
261      <t>
262        A Virtual Router operating in unicast mode <bcp14>MUST</bcp14> be
263        configured with one or more Unicast Peers. A configuration that
264        enables unicast mode without at least one peer is invalid, and the
265        Virtual Router <bcp14>MUST NOT</bcp14> operate in unicast mode until
266        corrected.
267      </t>
268      <t>
269        Each VRRP Router participating in a unicast-mode Virtual Router
270        <bcp14>MUST</bcp14> be configured with the addresses of all other
271        participating VRRP Routers for that Virtual Router and address family.
272      </t>
273      <t>
274        For IPv4 operation, each configured peer address <bcp14>MUST</bcp14>
275        be an IPv4 address that the receiving peer uses as the source address
276        for VRRP advertisements, as described in
277        <xref target="ipv4-fields"/>. For IPv6 operation, each configured peer
278        address <bcp14>MUST</bcp14> be an IPv6 link-local address used by the
279        receiving peer as the source address for VRRP advertisements, as
280        described in <xref target="ipv6-fields"/>.
281      </t>
282      <t>
283        The local router's own address <bcp14>MUST NOT</bcp14> appear in its
284        Unicast Peer List.
285      </t>

402        <name>Receiving VRRP Packets</name>
403        <t>
404          A VRRP Router operating a Virtual Router in unicast mode
405          <bcp14>MUST</bcp14> process only advertisements whose source address
406          matches an address in the configured Unicast Peer List for that
407          Virtual Router and address family.
408        </t>
409        <t>
410          A received VRRP packet for a unicast-mode Virtual Router
411          <bcp14>MUST</bcp14> be discarded if:
412        </t>

416          <li>the packet is received for the wrong address family, or</li>
417          <li>the packet is otherwise invalid according to <xref target="RFC9568"/>.</li>
418        </ul>
419        <t>
420          A VRRP Router operating a Virtual Router in unicast mode

426      <section anchor="host-facing-update">
427        <name>Host-Facing Behavior</name>
428        <t>
429          This document changes only advertisement delivery. The Active
430          Router's behavior with respect to the Virtual Router MAC address,
431          ARP, gratuitous ARP, IPv6 Neighbor Discovery, Router
432          Advertisements, Unsolicited Neighbor Advertisements, and forwarding
433          responsibility remains as specified in <xref target="RFC9568"/>.
434        </t>
435        <t>
436          In particular, unicast mode does not replace the Virtual Router MAC
437          with a multicast MAC. The Virtual Router MAC remains the well-known
438          unicast VRRP MAC associated with the VRID, as specified in
439          <xref target="RFC9568"/>.
440        </t>
441      </section>

444    <section anchor="operational">
445      <name>Operational Considerations</name>
446      <t>
447        A deployment using unicast mode <bcp14>SHOULD</bcp14> ensure that all
448        routers in a given Virtual Router are configured with a consistent
449        peer inventory. Inconsistent peer lists can create asymmetric
450        reachability and can lead to multiple routers independently deciding
451        that the Active Router has failed.
452      </t>
453      <t>
454        A deployment using unicast mode <bcp14>SHOULD</bcp14> continue to use
455        distinct priority values as recommended in <xref target="RFC9568"/> so
456        that Backup Routers do not transition to Active state simultaneously
457        after a failure.
458      </t>
459      <t>
460        This document does not update the VRRP management model. Future work
461        may standardize YANG or other management objects for peer lists,
462        source validation policy, and related unicast-mode configuration.
463      </t>

552    <section anchor="security">
553      <name>Security Considerations</name>
554      <t>
555        The security considerations of <xref target="RFC9568"/> continue to
556        apply. In particular, VRRP still provides no confidentiality and does
557        not prevent a hostile node from attempting to act as an Active Router.
558      </t>
559      <t>
560        Unicast mode introduces explicit peer configuration and requires that
561        a receiver validate the source address against the configured Unicast
562        Peer List. This provides an additional filtering mechanism beyond the
563        receive-side IPv4 TTL or IPv6 Hop Limit check.
564      </t>
565      <t>
566        Because this document preserves the requirement that the IPv4 TTL or
567        IPv6 Hop Limit be 255 on transmitted and accepted packets, the
568        protection against packets arriving from a remote network described in
569        <xref target="RFC5082"/> continues to apply.
570      </t>
571      <t>
572        A future specification for multi-hop unicast operation would need a
573        different security model and is outside the scope of this document.

      • <abstract>
        • <t>
          • The Virtual Router Redundancy Protocol (VRRP) is specified for multicast operation on a shared LAN in RFC 9568. Some deployments, including virtualized and cloud environments, require VRRP-like first-hop redundancy but cannot use multicast delivery for VRRP advertisements. This document updates RFC 9568 by defining an optional unicast mode for VRRP.
          • </t>
        • <t>
          • In unicast mode, VRRP advertisements are sent to configured peer addresses rather than to the VRRP multicast group. This document preserves the VRRP packet format, protocol number, virtual IP semantics, and host-facing forwarding behavior defined in RFC 9568, while adding explicit peer configuration and receive-side source validation for unicast operation.
          • </t>
        • </abstract>

      • <section anchor="intro">
        • <name>
          • Introduction
          • </name>
        • <t>
          • <xref target="RFC9568"/> specifies VRRP for IPv4 and IPv6 and assumes multicast operation on a shared LAN. In a number of modern deployments, redundant routers still need fast active/backup failover for a virtual default gateway, but the environment does not provide usable multicast support for VRRP advertisements.
          • </t>
        • <t>
          • The primary deployment driver is the continued need for the classic VRRP function of protecting a virtual IPv4 or IPv6 first-hop gateway in environments where multicast delivery is unavailable, undesirable, or operationally constrained. This includes virtualized, cloud, overlay, and other deployments in which Virtual Routers still provide a common host-facing gateway service, but control traffic between the Virtual Routers is exchanged through explicit peer connectivity instead of a simple multicast-capable LAN.
          • </t>
        • <t>
          • The intended use case for this document is not a generic active/backup role-election mechanism. Rather, it is a narrow extension of VRRP for deployments that want to preserve the familiar VRRP state machine, protocol number, virtual IP address semantics, Virtual Router MAC behavior, and host-facing forwarding model while replacing only the advertisement delivery method.
          • </t>

      • <section anchor="scope">
        • <name>
          • Scope and Applicability
          • </name>
        • <t>
          • This document updates <xref target="RFC9568"/> by defining an optional unicast mode of operation for VRRP. The unicast mode is intended for deployments that still want the classic VRRP model of a Virtual Router protecting one or more virtual IPv4 or IPv6 addresses, but that cannot rely on multicast delivery of advertisements.
          • </t>
        • <t>
          • The unicast mode defined here is limited to deployments in which the participating VRRP Routers can exchange advertisements without traversing a router that decrements the IPv4 TTL or IPv6 Hop Limit. This preserves the receive-side validation model of <xref target="RFC9568"/> and keeps the security and topology assumptions close to those of the base protocol.
          • </t>
        • <t>
          • This document does not define multi-hop operation. If a deployment requires routed multi-hop active/backup election or transport encapsulation other than IP protocol 112, that deployment is outside the scope of this specification.
          • </t>
        • </section>
      • <section anchor="usecases">
        • <name>
          • Use Cases and Deployment Drivers
          • </name>
        • <t>
          • The operational motivation for unicast VRRP is straightforward: operators still need the classic VRRP function of protecting a virtual IPv4 or IPv6 default gateway, but some modern deployments cannot rely on multicast delivery for VRRP advertisements.
          • </t>
        • <t>
          • Typical use cases include:
          • </t>
        • <ol spacing="normal" type="1">
          • <li>
            • <t>
              • Cloud and virtualized environments in which multicast is not available or is not exposed in a way that is operationally equivalent to a simple shared LAN.
              • </t>
            • </li>
          • <li>
            • <t>
              • Overlay or virtualized edge deployments in which the protected routers still present a common host-facing gateway service, but the control traffic between the routers is exchanged using explicit peer connectivity rather than a multicast-capable LAN.
              • </t>
            • </li>
          • <li>
            • <t>
              • Service-provider and data-center designs in which operators want to preserve the familiar VRRP packet format, state machine, and virtual IP semantics of <xref target="RFC9568"/> while replacing only the advertisement delivery method.
              • </t>
            • </li>
          • </ol>
        • <t>
          • This document addresses those use cases by defining a unicast delivery mode that stays close to the original VRRP model. It does not attempt to standardize broader routed active/backup election mechanisms that are no longer centered on protecting a virtual first-hop gateway.
          • </t>
        • </section>
      • <section anchor="definitions">
        • <name>
          • Additional Definitions
          • </name>
        • <dl newline="false" spacing="normal" indent="24">
          • <dt>
            • Unicast Mode
            • </dt>
          • <dd>
            • <t>
              • A mode of VRRP operation in which advertisements for a Virtual Router are sent as unicast IPv4 or IPv6 packets to configured peer addresses instead of to the VRRP multicast destination address.
              • </t>
            • </dd>
          • <dt>
            • Unicast Peer
            • </dt>
          • <dd>
            • <t>
              • A configured VRRP Router participating in the same Virtual Router and address family whose address is used as a permitted source and destination for unicast VRRP advertisements.
              • </t>
            • </dd>
          • <dt>
            • Unicast Peer List
            • </dt>
          • <dd>
            • <t>
              • The configured set of all other VRRP Routers that participate in a unicast-mode Virtual Router for a given address family.
              • </t>
            • </dd>
          • </dl>
        • </section>
      • <section anchor="overview">
        • <name>
          • Unicast VRRP Overview
          • </name>
        • <t>
          • A Virtual Router defined by this document operates in exactly one of two modes:
          • </t>
        • <ul spacing="normal">
          • <li>
            • multicast mode, as specified in <xref target="RFC9568"/>, or
            • </li>
          • <li>
            • unicast mode, as specified in this document.
            • </li>
          • </ul>
        • <t>
          • A VRRP Router operating a given Virtual Router in unicast mode <bcp14 NOT</bcp14> send VRRP advertisements for that Virtual Router to the VRRP multicast destination address. Instead, it <bcp14 send a copy of each advertisement to each address in the configured Unicast Peer List.
          • </t>
        • <t>
          • Except as updated by this document, the VRRP packet format, VRRP state machine, timer calculations, preemption behavior, Virtual Router semantics, virtual IP address behavior, and host-facing forwarding behavior remain as specified in <xref target="RFC9568"/>.
          • </t>
        • <t>
          • Unicast mode is configured per Virtual Router and per address family. Router. A VRRP Router <bcp14 NOT</bcp14> mix multicast mode and unicast mode for the same Virtual Router instance.
          • </t>
        • </section>
      • <section anchor="peer-config">
        • <name>
          • Peer Configuration
          • </name>
        • <t>
          • A Virtual Router operating in unicast mode <bcp14 be configured with one or more Unicast Peers. A configuration that enables unicast mode without at least one peer is invalid, and the Virtual Router <bcp14 NOT</bcp14> operate in unicast mode until corrected.
          • </t>
        • <t>
          • Each VRRP Router participating in a unicast-mode Virtual Router <bcp14 be configured with the addresses of all other participating VRRP Routers for that Virtual Router and address family.
          • </t>
        • <t>
          • For IPv4 operation, each configured peer address <bcp14 be an IPv4 address that the receiving peer uses as the source address for VRRP advertisements, as described in <xref target="ipv4-fields"/>. For IPv6 operation, each configured peer address <bcp14 be an IPv6 link-local address used by the receiving peer as the source address for VRRP advertisements, as described in <xref target="ipv6-fields"/>.
          • </t>
        • <t>
          • The local router's own address <bcp14 NOT</bcp14> appear in its Unicast Peer List.
          • </t>
        • </section>
      • <section anchor="rfc9568-updates">
        • <name>
          • Updates to RFC 9568
          • </name>
        • <section anchor="overview-update">
          • <name>
            • VRRP Overview
            • </name>
          • <t>
            • The references to multicast-only operation in <xref target="RFC9568" section="3" sectionFormat="of"/> are updated to allow an advertisement to be delivered either to the VRRP multicast destination address, as specified in <xref target="RFC9568"/>, or to configured Unicast Peers, as specified in this document.
            • </t>
          • </section>
        • <section anchor="protocol-update">
          • <name>
            • Protocol Processing
            • </name>
          • <t>
            • <xref target="RFC9568" section="5" sectionFormat="of"/> is updated so that a Virtual Router operating in unicast mode sends and receives VRRP advertisements only through the configured Unicast Peer List for that Virtual Router and address family.
            • </t>
          • </section>
        • <section anchor="ipv4-fields">
          • <name>
            • IPv4 Field Descriptions
            • </name>
          • <t>
            • For a Virtual Router operating in unicast mode, the IPv4 field descriptions in <xref target="RFC9568" section="5.1.1" sectionFormat="of"/> are updated as follows:
            • </t>
          • <ol spacing="normal" type="1">
            • <li>
              • <t>
                • The IPv4 source address <bcp14 be the primary IPv4 address of the sending interface, as specified in <xref target="RFC9568"/>.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv4 destination address <bcp14 be the IPv4 address of the configured Unicast Peer to which the copy of the advertisement is being sent.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv4 TTL <bcp14 be set to 255, and a received packet whose IPv4 TTL is not 255 <bcp14 be discarded.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv4 Protocol field <bcp14 remain 112.
                • </t>
              • </li>
            • </ol>
          • </section>
        • <section anchor="ipv6-fields">
          • <name>
            • IPv6 Field Descriptions
            • </name>
          • <t>
            • For a Virtual Router operating in unicast mode, the IPv6 field descriptions in <xref target="RFC9568" section="5.1.2" sectionFormat="of"/> are updated as follows:
            • </t>
          • <ol spacing="normal" type="1">
            • <li>
              • <t>
                • The IPv6 source address <bcp14 be the link-local address of the sending interface, as specified in <xref target="RFC9568"/>.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv6 destination address <bcp14 be the configured IPv6 link-local address of the Unicast Peer to which the copy of the advertisement is being sent.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv6 Hop Limit <bcp14 be set to 255, and a received packet whose Hop Limit is not 255 <bcp14 be discarded.
                • </t>
              • </li>
            • <li>
              • <t>
                • The IPv6 Next Header field <bcp14 remain 112.
                • </t>
              • </li>
            • </ol>
          • </section>
        • <section anchor="tx-update">
          • <name>
            • Transmitting VRRP Packets
            • </name>
          • <t>
            • <xref target="RFC9568" section="7.2" sectionFormat="of"/> is updated so that a VRRP Router operating a Virtual Router in unicast mode sends one copy of each VRRP advertisement to each configured Unicast Peer instead of sending the advertisement to the VRRP multicast group.
            • </t>
          • <t>
            • Other than the destination address, the packet contents <bcp14 be the same for each transmitted copy.
            • </t>
          • </section>
        • <section anchor="rx-update">
          • <name>
            • Receiving VRRP Packets
            • </name>
          • <t>
            • A VRRP Router operating a Virtual Router in unicast mode <bcp14 process only advertisements whose source address matches an address in the configured Unicast Peer List for that Virtual Router and address family.
            • </t>
          • <t>
            • A received VRRP packet for a unicast-mode Virtual Router <bcp14 be discarded if:
            • </t>
          • <ul spacing="normal">
            • <li>
              • the source address is not a configured Unicast Peer,
              • </li>
            • <li>
              • the IPv4 TTL or IPv6 Hop Limit is not 255,
              • </li>
            • <li>
              • the packet is received for the wrong address family, or
              • </li>
            • <li>
              • the packet is otherwise invalid according to <xref target="RFC9568"/>.
              • </li>
            • </ul>
          • <t>
            • A VRRP Router operating a Virtual Router in unicast mode <bcp14 ignore VRRP advertisements for that same Virtual Router received through multicast delivery.
            • </t>
          • </section>
        • <section anchor="host-facing-update">
          • <name>
            • Host-Facing Behavior
            • </name>
          • <t>
            • This document changes only advertisement delivery. The Active Router's behavior with respect to the Virtual Router MAC address, ARP, gratuitous ARP, IPv6 Neighbor Discovery, Router Advertisements, Unsolicited Neighbor Advertisements, and forwarding responsibility remains as specified in <xref target="RFC9568"/>.
            • </t>
          • <t>
            • In particular, unicast mode does not replace the Virtual Router MAC with a multicast MAC. The Virtual Router MAC remains the well-known unicast VRRP MAC associated with the VRID, as specified in <xref target="RFC9568"/>.
            • </t>
          • </section>
        • </section>
      • <section anchor="operational">
        • <name>
          • Operational Considerations
          • </name>
        • <t>
          • A deployment using unicast mode <bcp14 ensure that all routers in a given Virtual Router are configured with a consistent peer inventory. Inconsistent peer lists can create asymmetric reachability and can lead to multiple routers independently deciding that the Active Router has failed.
          • </t>
        • <t>
          • A deployment using unicast mode <bcp14 continue to use distinct priority values as recommended in <xref target="RFC9568"/> so that Backup Routers do not transition to Active state simultaneously after a failure.
          • </t>
        • <t>
          • This document does not update the VRRP management model. Future work may standardize YANG or other management objects for peer lists, source validation policy, and related unicast-mode configuration.
          • </t>
        • </section>

      • <section anchor="security">
        • <name>
          • Security Considerations
          • </name>
        • <t>
          • The security considerations of <xref target="RFC9568"/> continue to apply. In particular, VRRP still provides no confidentiality and does not prevent a hostile node from attempting to act as an Active Router.
          • </t>
        • <t>
          • Unicast mode introduces explicit peer configuration and requires that a receiver validate the source address against the configured Unicast Peer List. This provides an additional filtering mechanism beyond the receive-side IPv4 TTL or IPv6 Hop Limit check.
          • </t>
        • <t>
          • Because this document preserves the requirement that the IPv4 TTL or IPv6 Hop Limit be 255 on transmitted and accepted packets, the protection against packets arriving from a remote network described in <xref target="RFC5082"/> continues to apply.
          • </t>
        • <t>
          • A future specification for multi-hop unicast operation would need a different security model and is outside the scope of this document.
          • </t>

65    <abstract>
66      <t>
67        The Virtual Router Redundancy Protocol (VRRP) is specified for
68        multicast operation on a shared LAN in RFC 9568. Some deployments,
69        including virtualized and cloud environments, require VRRP-like
70        first-hop redundancy but cannot use multicast delivery for VRRP
71        advertisements. This document updates RFC 9568 by defining an
72        optional unicast mode for VRRP.
73      </t>
74      <t>
75        In unicast mode, VRRP advertisements are sent to configured peer
76        addresses rather than to the VRRP multicast group. This document
77        preserves the VRRP packet format, protocol number, virtual IP
78        semantics, and host-facing forwarding behavior defined in RFC 9568,
79        while adding explicit peer configuration and receive-side source
80        validation for unicast operation.
81      </t>
82    </abstract>

86    <section anchor="intro">
87      <name>Introduction</name>
88      <t>
89        <xref target="RFC9568"/> specifies VRRP for IPv4 and IPv6 and assumes
90        multicast operation on a shared LAN. In a number of modern
91        deployments, redundant routers still need fast active/backup failover
92        for a virtual default gateway, but the environment does not provide
93        usable multicast support for VRRP advertisements.
94      </t>
95      <t>
96        The primary deployment driver is the continued need for the classic
97        VRRP function of protecting a virtual IPv4 or IPv6 first-hop gateway
98        in environments where multicast delivery is unavailable, undesirable,
99        or operationally constrained. This includes virtualized, cloud,
100        overlay, and other deployments in which Virtual Routers still provide
101        a common host-facing gateway service, but control traffic between the
102        Virtual Routers is exchanged through explicit peer connectivity
103        instead of a simple multicast-capable LAN.
104      </t>
105      <t>
106        The intended use case for this document is not a generic active/backup
107        role-election mechanism. Rather, it is a narrow extension of VRRP for
108        deployments that want to preserve the familiar VRRP state machine,
109        protocol number, virtual IP address semantics, Virtual Router MAC
110        behavior, and host-facing forwarding model while replacing only the
111        advertisement delivery method.
112      </t>

129    <section anchor="scope">
130      <name>Scope and Applicability</name>
131      <t>
132        This document updates <xref target="RFC9568"/> by defining an
133        optional unicast mode of operation for VRRP. The unicast mode is
134        intended for deployments that still want the classic VRRP model of a
135        Virtual Router protecting one or more virtual IPv4 or IPv6 addresses,
136        but that cannot rely on multicast delivery of advertisements.
137      </t>
138      <t>
139        The unicast mode defined here is limited to deployments in which the
140        participating VRRP Routers can exchange advertisements without
141        traversing a router that decrements the IPv4 TTL or IPv6 Hop Limit.
142        This preserves the receive-side validation model of
143        <xref target="RFC9568"/> and keeps the security and topology
144        assumptions close to those of the base protocol.
145      </t>
146      <t>
147        This document does not define multi-hop operation. If a deployment
148        requires routed multi-hop active/backup election or transport
149        encapsulation other than IP protocol 112, that deployment is outside
150        the scope of this specification.
151      </t>
152    </section>

154    <section anchor="usecases">
155      <name>Use Cases and Deployment Drivers</name>
156      <t>
157        The operational motivation for unicast VRRP is straightforward:
158        operators still need the classic VRRP function of protecting a virtual
159        IPv4 or IPv6 default gateway, but some modern deployments cannot rely
160        on multicast delivery for VRRP advertisements.
161      </t>
162      <t>
163        Typical use cases include:
164      </t>
165      <ol spacing="normal" type="1">
166        <li>
167          <t>
168            Cloud and virtualized environments in which multicast is not
169            available or is not exposed in a way that is operationally
170            equivalent to a simple shared LAN.
171          </t>
172        </li>
173        <li>
174          <t>
175            Overlay or virtualized edge deployments in which the protected
176            routers still present a common host-facing gateway service, but the
177            control traffic between the routers is exchanged using explicit
178            peer connectivity rather than a multicast-capable LAN.
179          </t>
180        </li>
181        <li>
182          <t>
183            Service-provider and data-center designs in which operators want to
184            preserve the familiar VRRP packet format, state machine, and
185            virtual IP semantics of <xref target="RFC9568"/> while replacing
186            only the advertisement delivery method.
187          </t>
188        </li>
189      </ol>
190      <t>
191        This document addresses those use cases by defining a unicast delivery
192        mode that stays close to the original VRRP model. It does not attempt
193        to standardize broader routed active/backup election mechanisms that
194        are no longer centered on protecting a virtual first-hop gateway.
195      </t>
196    </section>

227    <section anchor="overview">
228      <name>Unicast VRRP Overview</name>
229      <t>
230        A Virtual Router defined by this document operates in exactly one of
231        two modes:
232      </t>
233      <ul spacing="normal">
234        <li>multicast mode, as specified in <xref target="RFC9568"/>, or</li>
235        <li>unicast mode, as specified in this document.</li>
236      </ul>
237      <t>
238        A VRRP Router operating a given Virtual Router in unicast mode
239        <bcp14>MUST NOT</bcp14> send VRRP advertisements for that Virtual
240        Router to the VRRP multicast destination address. Instead, it
241        <bcp14>MUST</bcp14> send a copy of each advertisement to each address
242        in the configured Unicast Peer List.
243      </t>
244      <t>
245        Except as updated by this document, the VRRP packet format, VRRP
246        state machine, timer calculations, preemption behavior, Virtual Router
247        semantics, virtual IP address behavior, and host-facing forwarding
248        behavior remain as specified in <xref target="RFC9568"/>.
249      </t>
250      <t>
251        Unicast mode is configured per Virtual Router.
252        A VRRP Router <bcp14>MUST NOT</bcp14> mix multicast mode and unicast
253        mode for the same Virtual Router instance.
254      </t>
255    </section>

257    <section anchor="peer-config">
258      <name>Peer Configuration</name>
259      <t>
260        A Virtual Router operating in unicast mode <bcp14>MUST</bcp14> be
261        configured with one or more Unicast Peers. A configuration that
262        enables unicast mode without at least one peer is invalid, and the
263        Virtual Router <bcp14>MUST NOT</bcp14> operate in unicast mode until
264        corrected.
265      </t>
266      <t>
267        Each VRRP Router participating in a unicast-mode Virtual Router
268        <bcp14>MUST</bcp14> be configured with the addresses of all other
269        participating VRRP Routers for that Virtual Router and address family.
270      </t>
271      <t>
272        For IPv4 operation, each configured peer address <bcp14>MUST</bcp14>
273        be an IPv4 address that the receiving peer uses as the source address
274        for VRRP advertisements, as described in
275        <xref target="ipv4-fields"/>. For IPv6 operation, each configured peer
276        address <bcp14>MUST</bcp14> be an IPv6 link-local address used by the
277        receiving peer as the source address for VRRP advertisements, as
278        described in <xref target="ipv6-fields"/>.
279      </t>
280      <t>
281        The local router's own address <bcp14>MUST NOT</bcp14> appear in its
282        Unicast Peer List.
283      </t>

400        <name>Receiving VRRP Packets</name>
401        <t>
402          A VRRP Router operating a Virtual Router in unicast mode
403          <bcp14>MUST</bcp14> process only advertisements whose source address
404          matches an address in the configured Unicast Peer List for that
405          Virtual Router and address family.
406        </t>
407        <t>
408          A received VRRP packet for a unicast-mode Virtual Router
409          <bcp14>MUST</bcp14> be discarded if:
410        </t>
411        <ul spacing="normal">
412          <li>the source address is not a configured Unicast Peer,</li>
413          <li>the IPv4 TTL or IPv6 Hop Limit is not 255,</li>
414          <li>the packet is received for the wrong address family, or</li>
415          <li>the packet is otherwise invalid according to <xref target="RFC9568"/>.</li>
416        </ul>
417        <t>
418          A VRRP Router operating a Virtual Router in unicast mode
419          <bcp14>MUST</bcp14> ignore VRRP advertisements for that same Virtual
420          Router received through multicast delivery.
421        </t>

425        <name>Host-Facing Behavior</name>
426        <t>
427          This document changes only advertisement delivery. The Active
428          Router's behavior with respect to the Virtual Router MAC address,
429          ARP, gratuitous ARP, IPv6 Neighbor Discovery, Router

431          responsibility remains as specified in <xref target="RFC9568"/>.
432        </t>
433        <t>
434          In particular, unicast mode does not replace the Virtual Router MAC
435          with a multicast MAC. The Virtual Router MAC remains the well-known
436          unicast VRRP MAC associated with the VRID, as specified in
437          <xref target="RFC9568"/>.

443      <name>Operational Considerations</name>
444      <t>
445        A deployment using unicast mode <bcp14>SHOULD</bcp14> ensure that all
446        routers in a given Virtual Router are configured with a consistent
447        peer inventory. Inconsistent peer lists can create asymmetric
448        reachability and can lead to multiple routers independently deciding
449        that the Active Router has failed.
450      </t>
451      <t>
452        A deployment using unicast mode <bcp14>SHOULD</bcp14> continue to use
453        distinct priority values as recommended in <xref target="RFC9568"/> so
454        that Backup Routers do not transition to Active state simultaneously
455        after a failure.
456      </t>
457      <t>
458        This document does not update the VRRP management model. Future work
459        may standardize YANG or other management objects for peer lists,
460        source validation policy, and related unicast-mode configuration.
461      </t>
462    </section>

464    <section anchor="implementation-status">
465      <name>Implementation Status</name>
466      <t>
467        This section records publicly documented implementation experience for
468        the benefit of reviewers. It is not part of the protocol
469        specification.
470      </t>
471      <t>
472        Public documentation reviewed by the authors shows that unicast VRRP
473        support already exists in multiple products and open-source
474        implementations, although not all of them match the protocol profile
475        defined in this document.
476      </t>
477      <t>
478        One implementation detail that varies across products is the
479        interaction between unicast advertisement delivery and Virtual Router
480        MAC behavior. Implementations that remain close to classic VRRP tend
481        to preserve the standard unicast Virtual Router MAC and change only
482        the control-packet delivery method, while more divergent "unicast
483        VRRP" modes move away from the classic virtual-gateway and neighbor
484        handling model.

550    <section anchor="security">
551      <name>Security Considerations</name>
552      <t>
553        The security considerations of <xref target="RFC9568"/> continue to
554        apply. In particular, VRRP still provides no confidentiality and does
555        not prevent a hostile node from attempting to act as an Active Router.
556      </t>
557      <t>
558        Unicast mode introduces explicit peer configuration and requires that
559        a receiver validate the source address against the configured Unicast
560        Peer List. This provides an additional filtering mechanism beyond the
561        receive-side IPv4 TTL or IPv6 Hop Limit check.
562      </t>
563      <t>
564        Because this document preserves the requirement that the IPv4 TTL or
565        IPv6 Hop Limit be 255 on transmitted and accepted packets, the
566        protection against packets arriving from a remote network described in
567        <xref target="RFC5082"/> continues to apply.
568      </t>
569      <t>
570        A future specification for multi-hop unicast operation would need a
571        different security model and is outside the scope of this document.
572      </t>
